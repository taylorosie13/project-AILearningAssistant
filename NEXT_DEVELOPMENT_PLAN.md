# 后续开发计划（按优先级）

## 1. 计划目标

基于当前项目进度，下一阶段开发遵循以下原则：

1. 先补齐“多模态学习助手”的核心能力，而不是优先做大规模重构。
2. 先保证功能闭环和稳定可演示，再做结构优化和体验增强。
3. 优先完成最接近项目蓝图核心价值的功能：音频、文档、多模态统一输入。

当前判断：

- 聊天、多轮上下文、图片上传、知识卡片管理已经具备可用基础。
- 音频/录音链路已经形成可演示闭环：iOS 端可录音、实时转写、保存本地语音内容、将音频附件带回聊天并上传到后端。
- PDF/文档理解链路已经具备基础能力，但仍需进一步联调和边缘样本验证。
- 启动引导、权限检查和局域网访问本地后端已经开始接入，但这部分还在收尾，属于当前最影响真机演示体验的短板。
- 项目当前更适合先把启动体验和关键联调跑稳，再继续做大范围架构调整。

---

## 2. 优先级规划

### P0：启动体验与核心多模态闭环

这是当前最高优先级，直接决定项目是否真正符合“多模态学习助手”的定位。

#### 2.1 启动引导与本地网络接入

当前状态：

- 应用启动时已经开始通过独立引导页统一检查关键权限和本地服务连接条件。
- iOS 端当前默认通过局域网地址访问本地 FastAPI 服务。
- 日志里已经能明确看到“本地网络权限被拒绝”导致无法访问本地后端的情况。

下一步重点：

- 完成启动引导页的交互收尾，让用户能明确知道卡在哪个权限或连接条件上。
- 验证本地网络权限弹窗、拒绝后的提示、重新尝试逻辑和设置页回跳提示。
- 确认真机与开发机同网段时，聊天、语音、文件上传都能稳定访问本地服务。

为什么这件事排最前面：

- 蓝图虽然强调的是多模态能力，但当前真机要稳定演示，前提是先能连上本地后端。
- 这一步没跑稳，后面的语音、文档、图片联调都会被卡住。

#### 2.2 音频/录音链路

当前状态：

- 已完成“录音 -> 实时转写 -> 本地保存 -> 加入聊天 -> 上传后端 -> 交给 Gemini”的主链路。
- 已支持录音页待处理文本编辑、已保存内容编辑、录音附件预览播放。
- 已修复语音工作台中的关键交互问题：双返回按钮、旧转写残留、误触自动发送。
- 已补齐基础异常提示，如权限不足、无可用麦克风输入、录音文件缺失等。

下一步重点：

- 做真机验证，重点检查不同麦克风输入设备下的录音稳定性。
- 验证录音文件格式与播放/上传兼容性，确保 `.m4a` 在真机上稳定可用。
- 验证“音频附件 + 文本问题 + 其他附件混用”的聊天体验。

#### 2.3 PDF/文档解析链路

当前状态：

- iOS 端支持选择 PDF 文件。
- 后端已支持 PDF 和主流 Office 文档上传，并可交给 Gemini 处理。
- 结果可在聊天界面中正常展示。

下一步重点：

- 用更多真实 PDF / PPT / DOCX 样本做联调。
- 重点验证大文件、图片型 PDF、复杂排版文档和转换失败场景。
- 收敛错误提示，让用户能明确区分“上传失败 / 转换失败 / 模型失败”。

#### 2.4 统一文件输入能力

当前状态：

- iOS 端已经具备统一文件选择入口，图片、音频、PDF 和主流 Office 文档都可复用同一上传链路。
- 后端已经具备统一的扩展名、MIME、大小和路径校验机制。

下一步重点：

- 在 iOS 端补充上传前大小检查，尽量在本地拦截超限文件。
- 继续验证多附件混合上传时的状态展示与失败重试是否稳定。

为什么优先做这一层：

- 当前项目已经有聊天壳子和图片能力，但音频和文档仍是蓝图缺口。
- 补齐这些能力后，项目才能从“学习场景聊天应用”升级为“真正的多模态学习助手”。

---

### P1：稳定性与联调

P0 完成后，立即进入稳定性补强阶段，避免多模态能力增加后整体不稳定。

#### 2.5 多场景联调

覆盖以下输入场景：

- 纯文本对话
- 图片上传解析
- 音频上传解析
- PDF 上传解析
- 启动引导进入主界面
- 真机通过局域网连接本地后端

重点检查：

- 会话切换时消息是否串线
- 删除会话时关联文件是否清理
- 卡片保存是否仍能关联正确会话
- 多种输入混用时是否有状态污染
- 语音工作台中的保存、编辑、回到聊天、再次录音之间是否有状态残留
- 真机录音与模拟器录音在权限、输入源和文件生成上的行为差异
- 启动页权限状态、主界面网络状态、聊天页错误提示三者是否一致

#### 2.6 错误处理补强

重点补强以下异常：

- API Key 无效或缺失
- 本地网络权限被拒绝
- 本机后端未启动或不在同一局域网
- 文件类型不支持
- 文件过大
- 上传超时
- 模型响应为空
- 数据库写入异常

交付标准：

- 前端能展示明确错误信息，而不是只在控制台打印。
- 后端返回统一、可读的错误描述。

#### 2.7 文件与资源保护

建议补充：

- 文件大小限制
- MIME 类型白名单
- 上传失败后的残留文件清理
- 请求超时控制
- 基础限流或防重复提交保护
- 真机录音链路日志补强，便于排查音频输入设备相关问题
- 本地网络探测日志补强，便于排查权限、DNS、地址配置和连通性问题

为什么排在这里：

- 多模态入口越多，联调复杂度越高。
- 这一阶段的目标不是新增很多功能，而是让已完成功能能稳定演示。

---

### P2：架构整理

当核心功能跑通且基本稳定后，再进行结构优化，降低后续维护成本。

#### 2.8 后端分层

当前问题：

- `backend/main.py` 承担了路由、Gemini 调用、文件处理、数据库访问等多项职责。

建议拆分：

- `routers/`
- `services/`
- `repositories/` 或 `db/`
- `utils/file_security.py`

目标：

- 让新增功能时不必持续堆积到 `main.py`。
- 提升代码可读性、可测试性和可维护性。

#### 2.9 iOS ViewModel 拆分

当前问题：

- `ChatViewModel.swift` 同时管理聊天、会话、卡片、错误提示，职责较重。

建议拆分方向：

- 聊天发送状态
- 会话管理状态
- 知识卡片状态

目标：

- 降低状态耦合。
- 为后续增加语音、文件、搜索等功能留下扩展空间。

#### 2.10 数据模型与错误模型统一

建议：

- 统一前后端字段命名规范。
- 统一错误返回结构。
- 为后续新增接口保留一致的扩展模式。

为什么不是最优先：

- 当前结构虽然偏大，但还能继续支撑一小段迭代。
- 在核心能力还没补齐前，架构重构的收益不如直接补功能高。

---

### P3：体验增强

这一阶段主要提升产品完成度和演示效果，建议在 P0、P1、P2 基本完成后推进。

#### 2.11 聊天与图片体验

- 聊天中的图片支持点击全屏预览
- 加强上传中、处理中、失败时的状态反馈
- 语音工作台按钮与文案进一步统一，减少“保存 / 加入聊天 / 编辑”之间的理解成本

#### 2.12 会话能力增强

- 历史会话搜索
- 会话预览内容优化

#### 2.13 知识卡片体验优化

- 分类筛选 chips
- 从卡片回跳原始会话
- 批量删除或批量调整分类

#### 2.14 渲染与交互优化

- 将阻塞式 `alert` 逐步升级为 toast 或 banner
- 减少 `MarkdownView` 对 CDN 的依赖，提升离线可用性

为什么放在最后：

- 这些功能会明显提升完成度，但不应先于核心能力与稳定性建设。

---

## 3. 推荐执行顺序

建议按 4 个迭代推进：

### 第 1 迭代

目标：先把启动、连接和核心能力验证到可稳定演示

- 完成启动引导页和权限检查收尾
- 真机验证局域网访问本地后端
- 真机验证音频/录音链路
- 联调 PDF / Office 文档链路
- 补齐 iOS 端上传前大小检查

### 第 2 迭代

目标：提升稳定性与可演示性

- 多场景联调
- 错误处理补强
- 文件大小、类型、超时等限制补充
- 优化启动页、本地网络权限提示和失败重试体验
- 优化语音工作台和聊天页之间的多模态混合输入体验

### 第 3 迭代

目标：为继续扩展做好结构准备

- 拆分后端 `main.py`
- 拆分 `ChatViewModel.swift`
- 统一数据模型与错误模型

### 第 4 迭代

目标：提升用户体验和演示效果

- 图片全屏预览
- 会话搜索
- 卡片筛选和回跳
- 非阻塞式反馈组件
- 离线渲染能力优化
- 语音工作台 UI 文案和操作路径精简

---

## 4. 当前阶段不建议优先做的事项

以下事项目前不建议排在前面：

1. 大规模 UI 美化
2. 复杂动画优化
3. 卡片批量功能优先于核心多模态能力
4. 在核心链路未补齐前进行彻底重构

原因：

- 这些事项虽然有价值，但不能直接提升项目与蓝图目标的匹配度。
- 当前最重要的是先让项目在“文本、图片、音频、PDF”四类输入上形成真正可用的学习闭环。

---

## 5. 阶段性目标总结

下一阶段最重要的不是“把项目做得更花”，而是让它更完整、更稳定、更符合最初定义。

最核心的推进顺序应当是：

1. 先把音频和 PDF 两条链路验证到稳定可演示
2. 做多场景联调和异常处理
3. 再进行后端与 iOS 端的结构整理
4. 最后打磨体验和展示效果

如果按这个顺序推进，项目会以更低风险进入一个“可演示、可继续扩展、也更接近毕业设计目标”的状态。
